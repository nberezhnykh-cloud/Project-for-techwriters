    Git Инструкция

    🐙 GIT

    🐙❓ Что такое GIT

    GIT – это распределённая система контроля версий, которая позволяет хранить разные версии файлов, отслеживать изменения и управлять ими.

    В архитектуре Git есть центральный (удалённый) репозиторий на сервере и локальные копии на устройствах участников проекта. Каждый работает в своей копии, при необходимости синхронизируя её с удалённым репозиторием для получения изменений с сервера или отправки своих локальных изменений на него.

    Такой подход обеспечивает высокую производительность системы (многие операции выполняются локально и не требуют сетевого подключения к удалённому серверу) и надёжность данных.

    ☁️🐙 Веб-сервисы для хранения репозиториев

    Для хранения и управления удалёнными git-репозиториями используются различные специальные веб-сервисы: GitLab, BitBucket, GitVerse, GitHub.

    GitHub – один из самых популярных таких сервисов, особенно для открытых (open-source) проектов.

    🐙⚙️ Как работает GIT

    ➕ Создание репозитория (GitHub)

    Ддя создания репозитория нужно выполнить несколько простых действий:

    1. Войдите в свой аккаунт GitHub;

    2. В правом верхнем углу нажмите кнопку [+] и выберите пункт «New repository»;

    3. Заполните поля в открывшейся форме:

      ✒️ Repository name – обязательное поле, название и slug вашего репозитория. Вы можете использовать латиницу, цифры и некоторые символы (точку, дефис и нижнее подчёркивание). Постарайтесь придумать название, которое будет отражать суть вашего репозитория;

      ✒️ Description – описание репозитория. Это поле необязательное, заполните его по своему усмотрению;

      ✒️ Public or Private. Публичный репозиторий будет виден всем пользователям Интернета (в режиме просмотра), а в приватном вы сами настраиваете список допущенных лиц;

      ✒️ Add a README file. Readme – это файл с описанием проекта;

      ✒️ Add .gitignore. В файле .gitignore содержится список неотслеживаемых гитом файлов и папок;

      ✒️ Choose a license. Лицензии защищают наше авторское право на создаваемый в репозитории продукт. Для учебы предлагаю установить стандартную универсальную общественную лицензию, которая разрешает всем желающим пользоваться вашими наработками. Раскройте список и выберите GNU General Public License;

    4. Нажмите кнопку Create repository.

    📦 Коммит (commit)

    GIT, как и любая система контроля версий, по сути является набором контрольных точек, каждая такая точка хранит информацию о файлах и изменениях, которые были внесены в эти файлы с момента создания предыдущей контрольной точки.

    При создании контрольной точки GIT как бы делает снимок проекта, запоминая, как выглядит каждый файл. Если файл с момента последнего снимка никак не изменился, GIT не будет его фиксировать ещё раз, а просто сошлётся на снимок с прошлой контрольной точки. Вся информация в системе хранится именно в виде таких снимков состояния файлов.

    Этот снимок называется коммит (commit), и он создаётся каждый раз, когда пользователь сохраняет внесенные изменения.

    🌿 Ветка (branch)

    Между коммитами можно свободно переключаться и просматривать разные версии проекта. Для навигации по истории коммитов, которых может быть огромное множество, в GIT есть специальные указатели.

    Изначально указатель стоит на первом коммите, а с каждым новым коммитом перемещается на него.

    Каждый новый указатель ответвляется от выбранного коммита и ведёт за собой новую цепочку коммитов. Такие указатели, а также цепочки коммитов, которые тянутся за ними, называются ветками (branch).

    В каждом проекте по умолчанию есть главная ветка, которую обычно называют master или main, которая содержит актуальную версию проекта (чистовик). Для внесения изменений следует создать новую ветку от главной (черновик) и работать в ней, чтоб случайно не внести несогласованные изменения в мастер-ветку.

    🔀 Слияние (Merge)

    После того, как черновик будет проверен, изменения из него можно перенесить в чистовик: влить ответвление обратно в мастер-ветку. Такая процедура в GIT называется слияние (merge).

    🐙⌨️ Полезные команды для работы

    Внутри проекта может быть создано сколько угодно веток – между ними можно свободно переключаться, объединять их друг с другом, переносить отдельные коммиты из одной ветки в другую и выполнять много других полезных операций. Всё это позволяет наладить командную работу и выполнять несколько задач параллельно.

    Все операции с файлами в GIT можно выполнять в терминале с помощью специальных команд.

    📥 Клонирование репозитория (команда git clone)

    Клонирование репозитория – это создание локальной копии репозитория на устройстве пользователя для дальнейшей работы с файлами репозитория:

    1. Откройте удалённый репозиторий в GitHub.
    2. На вкладке «Code» нажмите кнопку Code.
    3. В выпадающем окне перейдите на Local → HTML и скопируйте ссылку на репозиторий.
    4. Создайте или выберите на вашем устройстве папку, в которую вы хотите поместить локальный репозиторий.

    5. Откройте терминал и выполните команду для клонирования репозитория:

      git clone ссылка-на-репозиторий-из-пункта-3

      или откройте локальный репозиторий непосредственно через VS Code или другую программу.

    ⚙️ Настройка параметров конфигурации GIT (команда git config)

    Добавьте имя пользователя и электронный адрес:

    git config --global user.email e-mail@mail.ru
git config --global user.name "Anastasia.Savinkova"

    Этими данными по умолчанию будут подписываться все сделанные вами коммиты.

    Можно задать отдельные имя пользователя и электронную почту для определённого репозитория. Для этого необходимо перейти в локальный репозиторий и выполнить команды git config без флага --global:

    git config user.email new_e-mail@mail.ru
git config user.name "Ananastushok"

    👀 Просмотр текущего состояния репозитория (команда git status)

    Команда git status отображает текущую локальную ветку, состояние файлов в рабочей папке и в индексе: какие изменения ещё не добавлены в индекс, а какие уже готовы к коммиту.

    🔄 Обновление локального репозитория (команды git fetch и git status)

    Для работы с изменениями удалённого репозитория используются команды git fetch и git pull.

    Команда git fetch загружает последние изменения из удалённого репозитория, но не вливает их в локальные ветки, то есть не обновляет локальный репозиторий и не изменяет файлы в нём. Команду git fetch можно использовать для безопасного просмотра обновлений удалённого репозитория «без последствий», например, чтобы посмотреть новые ветки и проанализировать работу коллег.

    Команда git pull загружает последние изменения из удалённого репозитория и вливает их в локальные ветки, то есть обновляет состояние локального репозитория, синхронизируя его с удалённым репозиторием. Команду git pull используют в случаях, когда для работы необходимо использовать актуальную версию проекта.

    🔀 Переключение между ветками (команда git switch)

    Для переключения на другую ветку используется команда git switch название-ветки или git checkout название-ветки.

    🌿 Создание новой ветки (команда git branch)

    Как правило новая ветка создаётся для выполнения очередной задачи по обновлению проекта, поэтому ветка откалывается от главной ветки репозитория (main или master):

    1. Переключитесь на главную ветку репозитория:

      git switch master/main

    2. Заберите возможные обновления из удалённого репозитория:

      git pull

    3. Создайте новую ветку. По умолчанию ветка откалывается от текущей ветки (в нашем случае от master/main-ветки):

      git branch название_ветки

    4. Переключитесь на новую ветку:

      git switch название_ветки

    💾 Сохранение изменений

    Внесённые в файлы изменения находятся в рабочей директории. Перед созданием коммита необходимо добавить эти изменения в индекс – промежуточное пространство с метаданными об изменениях:

    1. Убедитесь, что изменения есть в рабочей директории:

      git status

      Непроиндексированные файлы указаны под заголовком Changes not staged for commit:

      git status
On branch RP

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   styles/reference.docx
        modified:   styles/styles.lua


no changes added to commit (use "git add" and/or "git commit -a")

    2. Добавьте изменения в индекс. Для этого выполните команду git add:

    3. Чтобы добавить один файл в индекс, укажите в команде git add полное название файла:

      git add styles/reference.docx

    4. Чтобы добавить в индекс все изменённые файлы, вместо названия файла в команде git add поставьте точку:

      git add .

      Если после git add . выполнить команду git status, можно увидеть, что проиндексированные изменения теперь отображаются под заголовком Changes to be commited:

      git add .
git status
On branch RP

Changes to be committed:
(use "git restore --staged <file>..." to unstage)
    modified:   styles/reference.docx

    5. Создайте коммит с помощью команды git commit -m "Описание коммита":

      git commit -m "Добавлена статья про GIT с полезными командами для работы"

    🚀 Публикация новой ветки

    Чтобы локальная ветка название_ветки появилась в удалённом репозитории и была доступна остальным участникам проекта, нужно создать её удалённую версию и связать их друг с другом.

    Для этого необходимо использовать команду git push с флагом --set-upstream:

    git push --set-upstream origin название_ветки

    В результате выполнения этой команды в удалённом репозитории origin создаётся ветка origin/название_ветки, которая связывается с вашей локальной веткой название_ветки. Теперь вы можете свободно выполнять команды git pull и git push в ветке название_ветки – GIT будет знать, из какой именно удалённой ветки забирать изменения при git pull и в какую удалённую ветку отправлять изменения при git push.

    📤 Отправка изменений на сервер

    Для отправки коммитов уже опубликованной ветки в удалённый репозиторий необходимо использовать команду git push.

    🔀 Слияние веток

    Для слияния двух веток необходимо использовать команду git merge:

    1. Переключитесь на целевую ветку (целевая – ветка, в которую будут вливаться изменения (main или master)):

      git switch master

    2. Выполните команду git merge название_сливаемой_ветки:

      git merge RP

    3. Отправьте изменения целевой ветки в удалённый репозиторий с помощью команды git push.

    📜 Просмотр истории изменений

    • Для просмотра истории коммитов выполните команду git log. В ответе отобразится список коммитов:

      git log
commit dfbf79b3bd0406845367feada5688ad8ffc7d85f (HEAD -> RP, origin/RP)
Author: Ananastushok <e-mail@mail.ru>
Date:   Tue May 19 16:19:36 2026 +0300

    Добавление файлов для документа РП

      Для просмотра следующих пунктов списка нажимайте клавишу Enter. Для выхода из команды git log нажмите клавишу q.

    • Для просмотра полной информации о конкретном коммите используйте команду git show с указанием хэша коммита:

      git show dfbf79b3bd0406845367feada5688ad8ffc7d85f
commit dfbf79b3bd0406845367feada5688ad8ffc7d85f (HEAD -> RP, origin/RP)

ddiff --git a/src/assets/images/pic1.png b/src/assets/images/pic1.png

    Что такое хэш коммита?

    Хэш коммита – это его уникальный идентификатор, состоящий из сорока символов (или его укороченная версия из семи символов).

    Хэш коммита можно посмотреть в выводе команды git log:

    git log
commit dfbf79b3bd0406845367feada5688ad8ffc7d85f # хэш коммита
Author: Ananastushok <e-mail@mail.ru>
Date:   Tue May 19 16:19:36 2026 +0300

    ↩️ Отмена изменений

    • Для отмены непроиндексированных изменений в файле используйте команду git restore название_файла. Изменения в файле будут безвозвратно удалены.

    • Для отмены проиндексированных изменений в файле используйте команду git restore --staged название-файла. Изменения в файле сохранятся, но будут удалены из индекса.

    • Для отмены локального коммита (который ещё не был отправлен в удалённый репозиторий) используйте команду git reset:

      git reset HEAD~1

      В результате выполнения этой команды указатель HEAD переместится на предыдущий коммит, а изменения из текущего коммита окажутся в рабочей директории.

    Важно!

    Не используйте команду git reset для коммитов, которые вы уже отправили на удалённый сервер.

    • Для отмены коммита, отправленного в удалённый репозиторий, используйте команду git revert хэш-коммита.

    Примечание

    Команда git revert не удаляет коммит, а отменяет внесённые в нём изменения и фиксирует это новым коммитом.

    📋 Просмотр веток

    Для просмотра всех веток локального репозитория используйте команду git branch.

    Для просмотра всех веток удалённого репозитория используйте команду git branch с флагом -r:

    git branch -r

    🗑️ Удаление веток

    • Локальная ветка

      Для удаления локальной ветки:

      1. Переключитесь на другую ветку.

      2. Выполните команду:

        git branch --delete название_ветки

    Важно!

    Команда git branch --delete удаляет только локальную версию ветки и не влияет на опубликованную в удалённом репозитории ветку, которую нужно удалить отдельно.

    • Удалённая ветка

      Для удаления удалённой ветки выполните команду:

      git push origin --delete название_ветки
